# План разработки `cf7-vk`

## Проблема и подход

Нужно создать WordPress-плагин по образцу `hokoo/cf7-telegram`, который отправляет уведомления Contact Form 7 не в Telegram, а в сообщения сообщества VK. Базовый каркас исходного плагина стоит сохранить: отдельные сущности транспорта, routing через channel, привязка форм к каналам, админка под CF7, React UI, REST-слой и `wpConnections`.

Главная архитектурная разница: в Telegram-версии входящие сообщения для первичной привязки чатов обнаруживаются через polling `getUpdates()` и фиксированную команду `/cf7tg_start`, а для VK в этом проекте нужно закладывать **Bots Long Poll API** и **настраиваемую команду первичной авторизации на уровне каждого VK-сообщества/бота**. Это значит, что переносить нужно не Telegram SDK-логику, а общую модель интеграции, заменив inbound/outbound слой и формат данных под VK.

## Решения, уже зафиксированные

- Использовать **VK Bots Long Poll API**, а не Callback API.
- Команда первичной авторизации настраивается **отдельно для каждого VK-сообщества/бота**.
- Сохраняем общую модель `bot/community -> channel -> form/chat`.
- Админский сценарий остаётся похожим на исходник: управление сообществами, каналами и привязками из подменю Contact Form 7.

## Эпики

### Эпик 1. Базовый каркас плагина и перенос общей архитектуры

Цель: перенести в новый репозиторий архитектурный скелет исходного плагина, но сразу в нейтральном или VK-ориентированном виде.

Состав:
- создать структуру WordPress-плагина (`plugin-dir`, bootstrap-файл, `composer.json`, `readme.txt`, `languages`, React-часть);
- перенести и адаптировать базовые сущности, коллекции, контроллеры, настройки, миграционный каркас;
- сохранить использование `wpConnections` для связей между сущностями;
- переименовать Telegram-специфичные идентификаторы, text domain, constants, route names и option keys в VK-совместимые.

Результат эпика:
- в репозитории есть рабочий каркас плагина, который устанавливается как самостоятельный продукт `cf7-vk`;
- сущности и REST/UI слой не содержат Telegram-зависимостей в именах и публичных интерфейсах.

### Эпик 2. VK transport layer: модель сообщества, API-клиент, отправка сообщений

Цель: заменить Telegram API на VK API для проверки доступа и отправки уведомлений.

Состав:
- спроектировать `Bot` как сущность VK-сообщества/бота, минимум с полями:
  - `access_token`,
  - `group_id`,
  - `api_version`,
  - `auth_command`,
  - служебные поля Long Poll (`server`, `key`, `ts`, время последней синхронизации, статус);
- реализовать VK API wrapper для:
  - проверки токена и группы,
  - получения Long Poll server data,
  - отправки сообщений через `messages.send`,
  - при необходимости получения сведений о диалогах/peer;
- сохранить паттерн хранения секрета через PHP constants как приоритет над БД;
- определить стабильный формат исходящего сообщения для VK, так как HTML/Markdown Telegram напрямую не переносимы.

Результат эпика:
- сообщество VK можно добавить в плагин, проверить его доступность и использовать для отправки сообщений.

### Эпик 3. Inbound flow для VK Long Poll и первичная авторизация по команде

Цель: переписать входящий сценарий обнаружения пользователей/диалогов под VK.

Состав:
- заменить Telegram polling `fetch_updates` на VK Long Poll polling;
- хранить и обновлять `server/key/ts`, корректно обрабатывать `failed=1/2/3`;
- отслеживать входящие `message_new` и связанные события разрешения сообщений сообщества, если они нужны для UX;
- внедрить на уровне каждого VK-бота настраиваемую команду первичной авторизации;
- при получении корректной команды:
  - нормализовать peer/user/conversation данные,
  - создать или обновить сущность чата/диалога,
  - связать её с соответствующим VK-ботом,
  - выставить начальный статус подключения;
- решить, будет ли команда сравниваться строго как текст сообщения или поддерживать варианты вроде `/start`, `!start`, `start`.

Результат эпика:
- пользователь VK может инициировать первичную привязку, отправив заданную администратором команду, после чего диалог становится доступным для дальнейшей маршрутизации.

### Эпик 4. Модель данных чатов/диалогов и routing-слой

Цель: адаптировать сущность `Chat` и связи под терминологию VK.

Состав:
- определить, что именно хранится как адресат:
  - private peer пользователя,
  - group chat peer,
  - conversation/peer id как универсальный идентификатор;
- обновить поля сущности `Chat` под VK-данные:
  - `peer_id`,
  - `from_id` или `user_id`,
  - `conversation_message_id` при необходимости,
  - `chat_type`,
  - имя/заголовок,
  - метаданные привязки;
- сохранить статусы подключения (`pending`, `active`, `muted`) либо упростить их после проверки UX;
- оставить каналы и связи form->channel, bot->channel, bot->chat, chat->channel, если они продолжают давать ту же ценность в VK-версии.

Результат эпика:
- routing-слой остаётся совместимым с исходной архитектурой, но опирается на корректные идентификаторы VK.

### Эпик 5. Адаптация обработки CF7-сообщений под формат VK

Цель: сохранить интеграцию с `wpcf7_before_send_mail`, но заменить форматирование и отправку.

Состав:
- сохранить текущую точку входа из CF7;
- оставить поддержку фильтров в духе исходника, если они ещё уместны;
- переписать преобразование HTML/Markdown в безопасный формат для VK:
  - либо plain text-first,
  - либо ограниченная конверсия ссылок и выделений в допустимый VK-формат;
- сохранить отправку в несколько каналов и несколько чатов внутри канала;
- пересмотреть обработку ошибок и логирование в контексте VK API.

Результат эпика:
- отправка уведомлений из CF7 работает через VK и не зависит от Telegram parse mode.

### Эпик 6. Admin UI и REST API под VK-сценарий

Цель: сохранить удобство исходной React-админки, но под другой onboarding flow.

Состав:
- обновить надписи, текстовый домен и структуру UI с Telegram на VK;
- заменить token-only форму создания бота на форму, где администратор задаёт:
  - название подключения,
  - `group_id`,
  - `access_token`,
  - `api_version` (с разумным default),
  - `auth_command`;
- заменить/переписать кнопки и API вокруг `ping`, `fetch_updates`, списка чатов и статусов;
- отобразить в UI:
  - команду авторизации для копирования,
  - состояние Long Poll,
  - обнаруженные диалоги,
  - их статусы и связи с каналами;
- локализовать все новые строки.

Результат эпика:

### Эпик 7. Миграции, документация, dev workflow

Цель: довести новый репозиторий до состояния полноценного отдельного продукта.

Состав:
- привести `composer`, npm, Docker/make-окружение и build scripts в состояние, соответствующее новому плагину;
- обновить `readme.txt`, корневой `README.md`, инструкции по настройке сообщества VK и Long Poll;
- описать обязательные настройки VK:
  - права токена,
  - включение сообщений сообщества,
  - включение Bots Long Poll и нужных событий;
- при необходимости упростить или временно отключить части migration-логики, если они относятся только к Telegram-наследию.

Результат эпика:

## Основные риски и замечания

### 1. Long Poll в WordPress-плагине

VK Bots Long Poll сам по себе ближе к фоновой интеграции, чем к типичному request/response WordPress-плагину. Надо заранее решить, как именно будет инициироваться polling:
- из админки через периодические REST-запросы, как в Telegram-версии;
- через WP-Cron;
- через гибридную схему.

Для первого релиза самым близким к исходнику выглядит polling из админки плюс сохранение `ts`, но это может означать, что обнаружение новых диалогов происходит только когда открыта страница настроек. Это не ломает отправку CF7 -> VK, но влияет на onboarding новых получателей.

### 2. Различие между `user_id`, `peer_id` и типами диалогов VK

В VK нельзя бездумно переносить понятие `chatID` из Telegram. На уровне модели надо рано выбрать единый идентификатор адресата, скорее всего `peer_id`, чтобы одинаково работать и с личными сообщениями, и с беседами.

### 3. Форматирование сообщения

Исходник умеет HTML/Markdown для Telegram. Для VK это нужно упростить и явно считать transport-specific formatter отдельным слоем, иначе CF7-контроллер останется слишком Telegram-центричным.

### 4. Пользовательское разрешение на сообщения сообщества

В VK отправка ответа зависит от того, разрешил ли пользователь сообщения сообществу или написал первым. Это нужно учесть в UX и в ошибках отправки: некоторые адресаты могут существовать в routing-слое, но быть фактически недоступными для исходящих сообщений.

### 5. Сильная связность имён и префиксов

В исходнике много Telegram-специфичных констант, хуков, route names и меток. Простая массовая замена опасна: лучше делать осознанное переименование и параллельно чистить публичные контракты.

## Заметки

- Решение про **Bots Long Poll API** уже принято, поэтому Callback API в базовый план не входит.
- Команда первичной авторизации должна быть видимой в UI и удобной для копирования.
- Если позже выяснится, что Long Poll в реальной эксплуатации неудобен для WordPress-плагина, ближайшая архитектурная альтернатива — перейти на Callback API, но сохранить остальной слой сущностей и routing без изменений.
